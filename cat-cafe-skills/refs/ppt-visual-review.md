# PPT 视觉审查 Gate — D 场景详细流程

> 来源：2026-04-04 三猫讨论（Ragdoll × Maine Coon × Siamese）
> 教训 1：D4 布局 bug（strategy-bar 吃 1fr）证明多猫视觉审查有效
> 教训 2：R3 信息密度腰斩事件 — 局部门禁（14px 字号）盖过页型愿景（华为高密战略页），三猫全部放行了退化版本

## 原则

**先审"对不对"（D1），再审"像不像"（D2）。**

D1 有 P1 未修 → 不进 D2。D2 的审美意见建立在布局正确的前提上。

**愿景优先 > 局部门禁。**

- reviewer 不能为了满足单一门禁（字号、留白、整洁）把页面改成另一种 archetype
- author 不能为了迎合 review 牺牲这页最核心的价值（信息密度、说服力、视觉冲击）
- 如果一条建议会动摇页型或核心价值，先讨论 archetype / 证据源 / 观看模式，不直接出执行型 P1

## 触发条件

- Slide HTML 完成（作者已自检截图看过一遍）
- 作者准备好 6 件套输入包

## 6 件套输入包

作者发起视觉审查时必须附带：

| # | 内容 | 说明 |
|---|------|------|
| 1 | 品牌+受众 brief | 目标公司、受众角色 |
| 1b | **页型（archetype）** | 华为高密战略页 / 发布会结论页 / KPI Dashboard / PDF handout 等 |
| 1c | **主观看模式** | presentation（大屏投影）/ document（PDF/打印）/ hybrid — **必须选主模式** |
| 2 | 本页目的 | 一句话：这页要让读者得到什么结论 |
| 3 | 截图/预览 URL | 浏览器渲染结果 |
| 4 | HTML/CSS 源码路径 | 定位布局 bug 的依据 |
| 5 | 密度数据 | whitespace%、element count、text nodes、overflow |
| 6 | 导出真相（如已导出） | native text / chart / table / screenshot fallback |

**硬门禁：缺任何一项 → 不进审，打回作者补齐。**

## 自动门禁（机器先跑，人再看）

D1/D2 之前先跑自动检查，能机器判的不用人眼：

- [ ] whitespace < 30%（density gate）
- [ ] overflow = 0（无截断/溢出）
- [ ] 正文字号 ≥ **页型矩阵**下限（见下表）
- [ ] 文字对比度 ≥ WCAG AA（4.5:1）
- [ ] 网格对齐偏差 ≤ 2px

**自动门禁全绿 → 进入 Archetype Guard → D1/D2 人工审查。有红 → 作者先修再提审。**

### 字号门禁：页型矩阵（不是单一数值！）

**页型决定字号，不是字号决定页型。**

| 页型 | 正文下限 | 辅助信息下限 | 典型场景 |
|------|---------|------------|---------|
| 发布会 / 对外 keynote | 14px | 12px | Steve Jobs 风、产品发布 |
| 华为高密战略页 | 10px | 8px | 内部战略分析、组件状态矩阵、行业对比 |
| KPI Dashboard | 12px（正文）| 9px（标签）| 数据看板、指标监控 |
| PDF handout / document | 9px | 8px | 会后阅读材料、打印稿 |

> **关键教训**：R3 事件证明"一刀切 14px"会逼迫作者砍内容来满足字号，最终把高密战略页改成低密结论页。
> 字号门禁必须跟着 archetype 走，reviewer 不能在不改 archetype 的情况下要求不匹配的字号。

## Archetype Guard（D1/D2 之前必做！）

**防止 review 把页面改型。** 这是 R3 密度腰斩事件后加的硬护栏。

1. 作者在 6 件套里声明本页 **archetype**（页型）
2. D1/D2 reviewer 在审查开始前先确认："我认同这个 archetype 吗？"
   - 认同 → 在该 archetype 框架内审查
   - 不认同 → **先和作者讨论 archetype 本身**，不要在错误的 archetype 下继续出 P1/P2
3. **Re-review 不得改变已定 archetype**，除非铲屎官拍板

**反面案例**：R2 Maine Coon给华为高密战略页出了"正文≥14px"P1 → 作者砍半内容满足字号 → 页型从"高密战略页"退化为"低密结论页" → 三猫无人拦截 → 铲屎官手动打回。

## D1: 布局/信息审查（QA/审查猫主导）

### 审查流程

1. 读 brief + archetype + 主观看模式 → 建立期望
2. 做 reviewer 自检三问（见下）
3. 看截图 → 第一印象：信息层级清不清楚？密度够不够？
4. 读 HTML/CSS → 检查布局结构是否正确（grid/flex 分配、overflow、background）
5. 看密度数据 → 对照 density gate 标准（30% whitespace 上限）
6. 出 P1/P2 清单

### Reviewer 自检三问（出单前必须过）

- [ ] 我这条意见是在保护既定 archetype，还是在偷偷改页型？
- [ ] 如果作者照做，这页的信息密度/说服力/视觉冲击会不会下降？
- [ ] 我是在解决"读者看不懂"的问题，还是只是在追求局部整洁？

**任何一题答案偏负面 → 不要直接出执行型 P1，先回到 Archetype Guard 讨论前提。**

### P1/P2 判定标准

| 级别 | 类型 | 定义 | 示例 |
|------|------|------|------|
| P1 | 布局 bug | CSS/HTML 导致的真实渲染错误 | strategy-bar 吃 1fr、overflow clipping |
| P1 | 信息失败 | 读者看完不知道重点是什么 | 标题和内容不匹配、层级混乱 |
| P1 | 密度失衡 | 该密不密（大块留白）或该疏不疏（挤到不可读） | 720px slide 只填 50%、文字重叠 |
| P1 | 页型漂移 | reviewer/author 的修改让页面从既定 archetype 漂到另一类页面 | 高密战略页被改成低密结论页 |
| P2 | 数据质疑 | 数据来源不明或可能过时 | 引用了 2023 年的数据标称"最新" |
| P2 | 受众错位 | 内容没错但不适合目标受众 | 给 CTO 看的 slide 用了太多技术术语 |

## D2: 审美/品牌审查（视觉把关猫主导）

### 审美五维检查清单

| 维度 | 检查项 |
|------|--------|
| 色彩体系 | 主色/强调色是否符合目标公司规范？对比度是否够（大屏投影要求高）？ |
| 字体排印 | 标题/正文层级跳跃够不够？行高在多行文本下是否易读？ |
| 空间网格 | 元素是否对齐网格？边距乘数是否和谐？ |
| 视觉元素 | 图标风格统一（线框 or 面性）？边框粗细一致？圆角统一？ |
| 密度平衡 | 高密度页是否用色块/卡片降噪？低密度页是否用图表/截图填充？ |

### 品牌审美滤镜

审查时根据目标品牌挂载对应滤镜：

- **华为**：红/黑/灰。模块化（Boxy）。粗体大字标语 + 分层架构图。严肃商务。
- **Apple**：极简。大面积留白。超大非衬线字体。每页只讲一件事。
- **阿里**：橙/科技蓝。Ant Design 体系。Dashboard 风格。数据驱动。

> 更多品牌滤镜：根据实际需求扩展此表。

### P1/P2 判定标准

| 级别 | 类型 | 定义 |
|------|------|------|
| P2 | 品牌偏移 | 不像目标公司的设计语言 |
| P2 | 视觉不一致 | 字号/卡片/边框/强调色/图标语言不统一 |
| P1 | 严重审美事故 | 配色完全错误（给华为用了苹果风）或对比度不可读 |

## 冲突仲裁（D1 vs D2 分歧）

当"好看"与"准确"冲突时（例：QA/审查猫要求加更多数据，视觉把关猫说太密了不好看），按受众优先级裁决：

| 受众 | 优先 | 理由 |
|------|------|------|
| CTO / 决策者 | 准确 > 美观 | 决策需要数据支撑 |
| 投资人 | 美观 ≈ 准确 | 信任感 = 专业度 + 视觉质感 |
| 技术团队 | 准确 > 美观 | 细节可读性优先 |
| 客户提案 | 美观 > 准确 | 第一印象决定是否继续听 |

**仲裁规则**：两方各陈述理由 → 按受众表裁决 → 无法裁决时升级铲屎官。

## 输出格式

审查完成后，reviewer 输出：

```markdown
## 视觉审查结果 — [slide 名称]

**品牌/受众**：[brief]
**审查员**：[猫名]（D1/D2）

### P1（必须修）
- [ ] [类型] 描述 — 根因 + 修复建议

### P2（建议修）
- [ ] [类型] 描述 — 建议

### 通过项
- [x] 密度 OK（whitespace X%）
- [x] 信息层级清晰
```

## 修复流程

作者收到审查结果后：

### Step 0: Author Synthesis（先取舍，再动手！）

收到 D1/D2 审查意见后，**不能直接全部照做**。必须先写一段取舍声明：

```
本页 archetype：[华为高密战略页]
采纳：[P1-xxx 布局 bug → 修]
不采纳：[P1-yyy 字号 14px → 因为本页是高密战略页，archetype 字号下限 10px]
理由：[...]
```

**反面案例**：R2→R3 作者把所有意见（14px、减密度、加结论块）都当成"必须满足"的硬约束，没做取舍，最终把页型改没了。

### Step 1-3: 修复 + Re-review

1. P1 必须全部修复
2. P2 可以技术论证不修，但必须给出理由
3. 修完后重新截图 + 更新密度数据 → 发起 re-review

### Regression Pair Gate（re-review 必做！）

每次 re-review，reviewer 必须做**前后版本对拍**：

1. 上一版截图 vs 当前版截图（并排比较）
2. 关键指标对比：

| 指标 | 上版 | 本版 | 趋势 |
|------|------|------|------|
| whitespace% | | | |
| element count | | | |
| text nodes | | | |
| facts retained | | | |
| 模块数 | | | |

3. **退化自检**：如果以下任何一项为真 → 自动 P1，打回重做：
   - 信息密度（text nodes）下降 > 20%
   - 模块数减少 > 30%
   - archetype 已实质改变（高密→低密、战略→结论等）

> **关键教训**：R3 事件中 reviewer 只看"新版自己是否成立"，没有对拍上一版。如果有对拍，"text nodes 从 49 减到 20+"会立刻触发 P1。
